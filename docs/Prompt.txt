====================================================

16. AUTHENTICATION & SUPABASE INTEGRATION
==========================================

Note: This feature adds user authentication via Supabase Auth.
It does NOT change the end-user data model or UI — all task/project
data still lives in localStorage. Supabase is used only for login sessions.

## Purpose

* Protect the app behind a login screen
* Support email/password authentication
* Enable future cloud sync (Supabase DB) without restructuring the frontend

## Credentials Management

Supabase credentials are NOT hardcoded in source files.

Instead:

* config.js — holds real credentials (gitignored, never committed)
* config.example.js — placeholder template committed to the repo

config.js shape:

```js
window.APP_CONFIG = {
  supabaseUrl: 'https://<your-ref>.supabase.co',
  supabaseKey: '<your-anon-key>'
};
```

app.js reads credentials from window.APP_CONFIG:

```js
const SUPABASE_URL = window.APP_CONFIG.supabaseUrl;
const SUPABASE_KEY = window.APP_CONFIG.supabaseKey;
```

If config.js is missing, the app shows a clear error page directing
the user to copy config.example.js and fill in their credentials.

## Login Screen

* Shown on app load when no authenticated session exists
* Fields: Email, Password
* Calls sb.auth.signInWithPassword()
* On success: hides #loginScreen, shows #app
* Guest mode: not supported — login is required to access the app

## Sign Out Button

* Location: sidebar footer, above the theme toggle
* Shown only when currentUser !== null
* Calls sb.auth.signOut(), sets currentUser = null, shows login screen

## Deployment & Secret Injection (GitHub Pages)

Since config.js is gitignored, GitHub Pages deployments use
GitHub Actions to inject credentials at build time:

1. Store SUPABASE_URL and SUPABASE_KEY as GitHub repository secrets
2. A GitHub Actions workflow (.github/workflows/deploy.yml) generates
   config.js from the secrets during the deploy job
3. The generated config.js is included in the Pages artifact but never
   committed to the repository

This means:
* Git history never contains credentials
* The deployed app receives credentials at runtime
* Anyone cloning the repo copies config.example.js to get started

## Team Member Modal (Add / Edit)

Replaced the old two-prompt flow (name + role) with a full modal overlay
matching the New Project modal pattern.

Modal fields:

* Avatar preset — grid of 12 emoji/icon options (auto-picks a color)
* Custom avatar URL — text input; overrides preset if set
* Color — dot color picker
* Name — text input (required)
* Role — text input with datalist suggestions
* Department / Team — text input
* Email — email input
* Phone — text input
* Skills — tag chip input

Avatar display priority on team cards:
1. avatarUrl set → <img> inside avatar bubble
2. avatarPreset set → renders preset icon
3. Fallback → colored-initials bubble

Extended user schema:

```js

  id, name, role, avatar, color, team,
  // added
  email: '',
  phone: '',
  avatarUrl: '',      // custom image URL; overrides preset
  avatarPreset: null, // index 0–11 into preset list
  skills: [],         // string[]
}
```

Each team card now shows:
* Email link (if set)
* Skills as small tag chips below the workload bar
* Edit button that opens the modal pre-populated with member data
