Row Level Security — Key Policies

workspacesSELECT
-- Members can see their workspaces
auth.uid() IN (
  SELECT user_id FROM workspace_members
  WHERE workspace_id = id
)
workspace_membersSELECT
-- See members of your workspaces
workspace_id IN (
  SELECT workspace_id FROM workspace_members
  WHERE user_id = auth.uid()
)
invitationsSELECT
-- Public token lookup (for invite page)
-- OR workspace member viewing invites
token IS NOT NULL -- anyone with token
OR workspace_id IN (
  SELECT workspace_id FROM workspace_members
  WHERE user_id = auth.uid()
)
tasksDELETE
-- Owner can delete any task
-- Member can only delete own tasks
workspace_id IN (members_subquery)
AND (
  get_my_role(workspace_id) = 'owner'
  OR created_by = auth.uid()
)
projects / tasks / milestones / sprintsALL OPS
-- All share the same workspace membership check
-- Helper function (run once in migration):
CREATE OR REPLACE FUNCTION is_workspace_member(
  _workspace_id uuid
) RETURNS boolean AS $$
  SELECT EXISTS (
    SELECT 1 FROM workspace_members
    WHERE workspace_id = _workspace_id
    AND user_id = auth.uid()
  );
$$ LANGUAGE sql SECURITY DEFINER;
task_activityALL OPS
-- Append-only: no UPDATE or DELETE allowed
-- SELECT / INSERT: workspace member only
-- UPDATE policy: FALSE (blocks all updates)
-- DELETE policy: FALSE (immutable log)
notificationsALL OPS
-- Users only see their own notifications
user_id = auth.uid()

-- INSERT: done via DB trigger on task events
-- (not by frontend directly)
⚠ Common RLS mistakes to avoid:
1. Forgetting to enable RLS on a table (data is public by default)
2. SELECT policies that don't scope to workspace_id — can leak cross-tenant data
3. Allowing INSERT on notifications from the frontend — use DB triggers instead
4. Not using SECURITY DEFINER on helper functions — can bypass RLS

storage.js — replaces the old Storage object

All Supabase DB operations live here. Each function maps to one table:
loadTasks(workspaceId), saveTask(task), deleteTask(id)
loadProjects(workspaceId), saveProject(project)
No more loading the entire state blob — every read/write is targeted.

state.js — single source of truth

Still a global state object, but now loaded from individual table reads rather than one blob. Mutations go through explicit functions (setState(), updateTask()) that also trigger a scheduleSave() to Supabase for that specific entity only.

permissions.js — permission guards

Role checked once at load, stored in state. Guards are synchronous:
canEdit(entityType) → true/false
canDelete(entity) → checks role + ownership
getMyRole() → 'owner' | 'member' | 'guest'
UI elements hide/disable based on these guards.

app.js becomes a thin entry point

Only handles: import all modules → init auth → check session → load workspace → set up realtime → render initial view → bind global keyboard shortcuts. No business logic. Target: under 200 lines.

Invite Link Flow (No Email Required)

1

Owner clicks "Invite Member"

Opens invite modal. Selects role (Member or Guest). Optionally adds a label (e.g. "For Zain — Developer").

2

Frontend generates token + inserts row

Calls crypto.randomUUID() for a secure token. Inserts into invitations table: token, workspace_id, role, expires_at (7 days).

3

Shareable link displayed

UI shows: https://zainknoman.github.io/TaskTracker/?invite=<token>
Owner copies it and sends via Slack, WhatsApp, email, etc.

4

Invitee opens link

App detects ?invite=token in URL. Fetches invitation row by token (public RLS). Shows workspace name + role. Prompts to log in or sign up.

5

After auth: accept invitation

Calls acceptInvitation(token, userId). Inserts into workspace_members. Sets invitation status to 'accepted'. Token is now consumed.

6

Redirect to workspace

URL param cleared. App loads the workspace. Member sees all projects and tasks. Role determines what they can do.

Realtime Strategy (Live Sync Only)

Channel: workspace-{workspace_id}

Single channel per workspace. Subscribes to changes on tasks, projects, milestones, sprints filtered by workspace_id.

Events: INSERTUPDATEDELETE
Handler: update local state → re-render affected view only (not full page).

Channel: user-{user_id}

Per-user channel for notifications. Subscribes to notifications table filtered by user_id.
Events: INSERT → update bell badge count + append to notification list.

Optimistic updates

Frontend updates local state immediately on user action (no waiting for DB confirmation). If the Supabase write fails, the state is rolled back and a toast error shown. The realtime event from the successful write is ignored if it matches local state (idempotent merge).

Not on Realtime: task_activity

task_activity is append-only. No realtime subscription — activity log is fetched on demand when a task detail modal opens. Avoids noisy channel events for every log insert.
